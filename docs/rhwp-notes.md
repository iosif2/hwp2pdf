# rhwp 조사 노트

**프로젝트**: hwp2pdf · **생성일**: 2026-04-23 · **수정일**: 2026-04-23

## 개요

rhwp는 Rust로 작성된 오픈소스 HWP/HWPX 파싱 및 렌더링 엔진이다.
HWP 문서를 내부 Document Model로 변환한 뒤 SVG 또는 PDF로 출력한다.

---

## 제약 사항

### PDF 출력 미안정

- PDF 출력 기능은 공식적으로 안정화 단계가 아님
- 내부적으로 SVG 페이지를 병합하여 PDF를 생성하는 방식
- 복잡한 레이아웃 문서에서 결과물 품질을 보장할 수 없음

### 레이아웃 깨짐

- 일부 문서에서 레이아웃 재현 실패 가능
- 특히 다단 구성, 겹친 객체, 복잡한 표 구조에서 발생 빈도 높음

### upstream 의존성

- rhwp는 외부 오픈소스 프로젝트 — upstream 변경에 따라 API 호환성이 깨질 수 있음
- 직접 호출부를 wrapper로 격리하여 upstream 변경 영향 범위를 최소화해야 함

## 폰트 이슈

### 의존 구조

rhwp의 렌더링 품질은 시스템에 설치된 폰트에 직접 의존한다.
폰트가 없거나 경로가 다르면 렌더링 결과가 환경마다 달라진다.

### 주요 증상

- 폰트 미설치 시 글자가 깨지거나 공백으로 출력
- 환경별 폰트 경로 차이로 인한 비결정적 렌더링
- 컨테이너 환경에서는 기본 한글 폰트가 존재하지 않는 경우가 많음

### 대응 방안

- 배포 환경에 한글 폰트를 명시적으로 포함
- Noto Sans KR / Noto Serif KR을 기본 폰트로 고정
- rhwp가 참조하는 폰트 경로에 직접 배치

```
/app/fonts/
    NotoSansKR-Regular.ttf
    NotoSerifKR-Regular.ttf
```

- 폰트 로딩 실패는 반드시 로그에 기록하고 즉시 에러로 처리

## 호출 방식 비교

### CLI 방식

```bash
rhwp export-pdf input.hwp -o output.pdf
```

- 프로세스 생성 오버헤드 있음
- 구현 단순, 초기 프로토타이핑에 적합
- 에러 핸들링이 exit code 기반으로 제한적

### 라이브러리 직접 호출 방식

```rust
let doc = HwpDocument::from_bytes(&bytes)?;
let pages = doc.render_to_svg_pages()?;
let pdf = svg_to_pdf(pages)?;
```

- 프로세스 생성 없음 → 낮은 레이턴시
- 메모리 내 처리 가능 → 임시 파일 최소화
- 에러를 Rust Result로 처리 가능
- **초기 구현에서 이 방식을 목표로 함**

## 참고

- 폰트 문제는 런타임 에러가 아닌 무음 렌더링 실패(silently broken output)로 나타날 수 있어 테스트 시 결과물 육안 확인 필요
- OCR 기반 fallback은 초기 스코프 밖
